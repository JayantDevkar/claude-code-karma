## Open Issues

### [CRITICAL] Subagent JSONL Files Never Have `end_turn` Stop Reason

**Severity**: High
**Impact**: Completion detection for subagents relies ENTIRELY on time-based heuristics

**Observation** (from meta-testing 2026-01-09 - Session 734272f9):
```
Subagent stop_reason analysis:
- a193a59: end_turn=0, null=9
- a223ed1: end_turn=0, null=6
- a521363: end_turn=0, null=16
- ae16cd4: end_turn=0, null=6
- ad30af0: end_turn=0, null=5

Total `end_turn` in subagent JSONL files: 0
Total `end_turn` in main session JSONL files: 3 (opus model only)
```

**Root Cause**: Claude Code's Task tool writes JSONL files where `stop_reason` is ALWAYS `null` for assistant messages. The `end_turn` stop reason only appears in:
1. Main session JSONL files (not subagents)
2. Only with opus model (observed pattern)

**Current State Detection Logic** (`subagent-watcher.ts:253-276`):
```typescript
// Priority 3: Check for 'completed' state
else if (hasStopReason && lastStopReason === 'end_turn') {  // NEVER FIRES for subagents!
  state = 'completed';
} else if (lastMessageType === 'assistant' && lastMessageHasTextOnly) {
  if (msSinceModified > 5000) {  // 5 seconds - THIS is the only path
    state = 'completed';
  }
}
```

**Implications**:
1. The `end_turn` check is dead code for subagents
2. All completion detection uses the 5-second file modification heuristic
3. If an agent finishes but the file is still being written (race condition), state may be wrong
4. The heuristic masks other bugs because it "happens to work" in most cases

**Recommended Fix**:
1. Remove reliance on `end_turn` for subagent completion detection
2. Consider alternative signals:
   - Presence of final text-only message without pending tool_use
   - Agent output file existing (Task tool writes to output file on completion)
   - Explicit completion marker from Claude Code (feature request)

**Problem**: A Claude Code session has no way to know its own session ID. The ID is:
1. Generated by Claude Code internally

**Questions**:
1. Can Claude Code accept `--session-id` flag to use a user-provided ID?

**Current Workaround**:
- Find session ID manually: `ls -t ~/.claude/projects/<encoded-path>/*.jsonl | head -1`
- Or parse from `git status` output showing modified JSONL paths

**Potential Fixes**:
1. **Auto-detect**: Radio commands could find most recent JSONL file in project directory
2. **Hooks integration**: Request Claude Code expose session ID to hook scripts
3. **Dashboard bridge**: Dashboard already knows session ID when started with `--session-id` flag

### [OPEN] State Inference False "Failed" Detection

**Severity**: Medium
**Impact**: Active/completed agents incorrectly shown as "failed"

**Observation** (from meta-testing 2026-01-09):
```
Actual task status: completed (all 4 doc subagents finished successfully)
Radio scan status:  2 completed, 2 failed (a19e094, ae635d5)
```

The state inference in `subagent-watcher.ts` marks agents as "failed" when:
- Any `tool_result` has `is_error: true` (even recoverable errors like file not found)
- The agent was actively working but encountered a single tool error during exploration

**Root Cause**: The `hasToolError` flag triggers "failed" state without considering:
1. Whether the error was fatal or recoverable
2. Whether the agent continued working after the error
3. The actual final stop_reason (which is always `null` anyway - see above)

**Potential Fix**: Only mark as "failed" when:
- `stop_reason === 'error'` (explicit error termination)
- Or file is stale AND last message has `is_error: true`
- Ignore transient tool errors if agent continues working

**Note**: This bug may be masked by the 5-second heuristic when agents complete quickly without errors. The previous test showing 50% false-failed rate occurred when agents encountered tool errors during exploration.

### [FIXED] TTL Mismatch - Session List vs Agent Status

**Severity**: Low
**Impact**: Stale agent IDs in session list after status expires

**Observation**:
```typescript
const TTL = {
  STATUS: 300000,           // 5 minutes
  SESSION_AGENTS: 3600000,  // 1 hour (12x longer)
};
```

Agent status can expire while ID remains in session list. Current code handles this by filtering nulls, but creates inconsistent counts.

**Solution**: Already implemented in `src/walkie-talkie/agent-radio.ts`
- `refreshSessionTTL()` method is called on every `setStatus()` update
- Keeps SESSION_AGENTS and parent children list TTLs refreshed while agents are active

- Filters agents with toolCount === 0 AND no assistant messages
- Returns null to exclude from agent list

### [FIXED] No Subagent Depth Tracking

**Severity**: Low
**Impact**: Can't distinguish nested subagent levels

**Observation**: JSONL files are flat - no indication if a subagent spawned another subagent.

The `parentId` in radio comes from the watching session, not the actual spawning agent. This means:
- All subagents appear as direct children of the main session
- True multi-level hierarchies (agent A spawns agent B spawns agent C) not captured

**Root Cause**: JSONL files don't record who spawned them, only the root session ID.

**Solution Implemented**: Depth field added to AgentStatus
- `depth` field in `AgentStatus` type (`src/walkie-talkie/types.ts:122`)
- Computed in `AgentRadioImpl` constructor (`src/walkie-talkie/agent-radio.ts`)
  - Session children: depth = 0
  - Agent children: depth = parent.depth + 1
- Added to `SubagentInfo` in subagent-watcher (default: 0 for JSONL-discovered agents)

**Limitation**: JSONL-discovered agents always have depth=0. True depth tracking requires agents to register via radio IPC.
