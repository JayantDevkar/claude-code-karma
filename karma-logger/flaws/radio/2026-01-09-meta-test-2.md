## Logic Observations

### [LOGIC-1] IPC agents and JSONL agents never merged

**list-agents output**: 5 agents (a40a8db, a658144, accd98c, ef646800..., main)

**scan output**: 4 agents (a01fa88, a023b52, aa260b6, ada453f)

These are completely disjoint sets. No overlap. Two separate tracking systems running independently.

**Impact**: tree command only shows IPC agents, not JSONL-discovered subagents.

---

### [LOGIC-2] False-positive failure detection (confirmed)

**Agent**: aa260b6

**Timeline**:
- Tool count grew: 11 -> 12 -> 19 -> 21 (actively working)
- Radio state: "failed" throughout
- TaskOutput status: "completed"
- JSONL error count: 2 (`is_error: true`)
- Error type: "EISDIR: illegal operation on a directory, read" (recoverable)

**Root cause**: Any `is_error: true` in JSONL triggers permanent "failed" state, even if agent continues working and completes successfully.

---

### [LOGIC-3] Agent with full session UUID as agentId

**Observed**: Agent ID `ef646800-0973-48d2-9e42-ec30a3542f80` exists in IPC agents list

This is the same as my session ID. Unclear why a full UUID appears as an agent ID when other agent IDs are 7-char hashes.

---

### [LOGIC-4] All IPC agents have null metadata

```json
{
  "agentType": "unknown",
  "model": "unknown",
  "parentId": null,
  "parentType": "session",
  "metadata": {}
}
```

No model info, no agent type, no parent-child relationships populated for IPC agents.

---

### [LOGIC-5] Main agent stuck in "pending" state

My "main" agent never transitioned to "active" state even though I was actively running commands. The `setStatus` was never called to update state.
